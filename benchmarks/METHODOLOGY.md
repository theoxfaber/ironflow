# IronFlow Benchmark Methodology

This document describes the exact methodology used to produce the benchmark numbers
published in the [IronFlow README](../README.md). All test scripts are open-sourced
here so the results can be independently verified and challenged.

---

## TL;DR Results

| Orchestrator | Time (50 tasks) | Peak Memory | vs. IronFlow |
|---|---|---|---|
| **IronFlow v0.1.0** | **0.08s** | **12 MB** | — |
| Dagster (latest) | 1.07s | 98 MB | 13× slower |
| Prefect (latest) | 4.96s | 155 MB | 62× slower |
| Apache Airflow (latest) | 31.38s | ~380 MB | 392× slower |

## What We Measured

**Orchestrator overhead only.** Each task in the pipeline runs:

```bash
echo 'Executing task N'
```

This is pure no-op work. The timing difference is 100% orchestrator overhead:
scheduling, state transitions, dependency resolution, persistence, and teardown.

---

## Pipeline Definition

**50 tasks, fully sequential** (each task depends on the previous one).

This is the worst case for an orchestrator — no parallelism is possible, so every
state write must complete before the next task can start. It maximally stresses
the scheduler loop and persistence layer.

### IronFlow (`ironflow_dag.toml`)
```toml
[dag]
id = "benchmark_dag_100"
description = "50-task sequential benchmark"

[[dag.tasks]]
id = "task_0"
operator = "bash"
[dag.tasks.config]
command = "echo 'Executing task 0'"

[[dag.tasks]]
id = "task_1"
operator = "bash"
depends_on = ["task_0"]
[dag.tasks.config]
command = "echo 'Executing task 1'"

# ... repeated for all 50 tasks
```

Generated by [`generate_dags.py`](./generate_dags.py).

### Apache Airflow (`airflow_home/dags/airflow_dag.py`)
```python
from airflow import DAG
from airflow.providers.standard.operators.bash import BashOperator
from datetime import datetime

with DAG('benchmark_dag_100', start_date=datetime(2023, 1, 1), schedule=None, catchup=False) as dag:
    prev = None
    for i in range(50):
        task = BashOperator(task_id=f'task_{i}', bash_command=f"echo 'Executing task {i}'")
        if prev:
            prev >> task
        prev = task
```

### Dagster (`dagster_dag.py`)
```python
from dagster import job, op

@op
def dummy_op(prev=None):
    return

@job
def benchmark_dag_100():
    prev = None
    for i in range(50):
        if prev is not None:
            prev = dummy_op.alias(f"task_{i}")(prev)
        else:
            prev = dummy_op.alias(f"task_{i}")()

if __name__ == "__main__":
    benchmark_dag_100.execute_in_process()
```

### Prefect (`competitor_dag.py`)
```python
from prefect import flow, task

@task
def dummy_task(i, prev_result=None):
    return i

@flow(name="benchmark_dag_100")
def benchmark_flow():
    prev = None
    for i in range(50):
        prev = dummy_task(i, prev_result=prev)

if __name__ == "__main__":
    benchmark_flow()
```

## How Timing Was Measured

All timings use Python's `time.time()` wall-clock measurement.

### IronFlow
1. Start the IronFlow release binary as a subprocess (`start --with-api`)
2. Wait 2 seconds for startup (this wait is **subtracted** from the final time)
3. Trigger the DAG via its HTTP API (`POST /api/dags/benchmark_dag_100/trigger`)
4. Poll the SQLite DB every 50ms until `dag_runs.status = 'success'`
5. Stop the clock and terminate the server
6. **Reported time = time from trigger to completion (startup cost excluded)**

### Airflow
Airflow was run with `airflow dags test benchmark_dag_100` — its single-process
in-process test mode. The Airflow log itself reports `run_duration=31.377815`.
This mode bootstraps a full FastAPI execution API and XCom HTTP server in-process,
which explains the overhead.

Timing was measured wall-clock from process start to exit.

### Dagster / Prefect
Timed from `subprocess.Popen()` to `process.returncode` being set.

---

## How Memory Was Measured

Peak RSS (Resident Set Size) was sampled every 10–50ms via `psutil.Process.memory_info().rss`
throughout the run. The reported value is the **maximum** RSS observed.

For Airflow's in-process mode, additional child processes were not spawned, so
the single process RSS captures the full footprint.

## How to Reproduce

### Prerequisites

```bash
# IronFlow (Rust)
cargo build --release

# Python venv for Prefect
python3 -m venv .venv
source .venv/bin/activate
pip install prefect psutil

# Python venv for Airflow + Dagster
python3 -m venv .venv_heavy
source .venv_heavy/bin/activate
pip install apache-airflow dagster psutil
```

### Run

```bash
cd benchmarks

# Generate all DAG files
.venv_heavy/bin/python generate_dags.py

# Run all benchmarks
.venv_heavy/bin/python benchmark_runner.py

# Results written to heavy_results.json
cat heavy_results.json
```

The benchmark script handles everything automatically:
- Generates the DAG files
- Starts IronFlow, triggers via API, polls until completion
- Runs Prefect and Dagster as subprocesses
- Runs Airflow in `dags test` mode

## Known Caveats

1. **Airflow was measured in `dags test` mode** — this is its fastest path but still
   boots a non-trivial internal FastAPI server. A production Airflow cluster would
   be materially slower.
2. **Results will vary** by hardware. The ratios should remain consistent; absolute
   numbers depend on CPU speed and I/O.
3. **Sequential workloads favour IronFlow most.** For massively parallel DAGs where
   task runtime dominates, the orchestrator overhead matters less.
4. **IronFlow's startup (2s) is excluded.** For a fair trigger-to-completion comparison,
   we only time from the HTTP trigger call to run completion.
